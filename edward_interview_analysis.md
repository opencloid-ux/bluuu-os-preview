## 1. Типичный день (структура работы)

### Утренняя смена:
1. Проверить сообщения по сегодняшним турам (транспорт, встречная точка)
2. Ввести OTA-букинги и ночные букинги (следующий день / послезавтра)
3. Проверить наличие лодок на следующий день → если нет — закрыть availability для shared/premium shared
4. Весь день: ответы на запросы гостей + ввод OTA как можно быстрее
5. В конце смены: проверка всех туров (особенно private), доп.опции, передача следующей смене

### Вечерняя смена:
1. Проверить просроченные задачи с прошлого дня
2. Убедиться что весь транспорт забронирован
3. Сообщить G. David и водителям о last-minute букингах
4. Подтвердить все vendor boats с информацией о гостях
5. Продолжать чаты

### Использование инструментов:
- **Kommo (Como):** ~основной инструмент весь день
- **WhatsApp (GR group, G. David, Dima, Karim):** ~20% времени
- **Odoo:** ~10% — только при новых букингах / добавлении транспорта и extras
- **Google Maps:** проверить где находится гость → рекомендовать shuttle point
- **ChatGPT:** при сложных / агрессивных гостях (подобрать правильные слова)

## 2. Процесс работы с inquiries

- **Открытие:** шаблонное приветствие («Hi, I'm Edward, how can I help?»)
  - Дорабатывает вручную если шаблон не подходит под конкретный контекст
- **Квалификация:** сколько человек + дата → private или shared + рекомендация лодки
- **Booking link:** отправляет ссылку с сайта → гость бронирует → подтверждение
- **Follow-up задачи:**
  - Не забронировал через день → задача каждый день
  - Дата через неделю → задача каждый день
  - Дата далеко → каждые 2-3 дня

## 3. Топ-5 вопросов гостей

1. Что по цене? Лучшая цена?
2. Есть ли тур на конкретную дату?
3. Есть ли вегетарианское меню?
4. Трансфер включён? (частое непонимание с OTA)
5. Какая погода?

---

## 4. Информация, которой не хватает

- **Спецификации лодок:** год, размер, двигатель, скорость — раньше были на старом сайте, теперь нет → нельзя ответить гостю быстро
- **Кастомный итинерари:** нужно уточнять у Geebran
- **Погода в Сарангане в 6:30:** нет живого источника → ждут пока кто-то из команды в office/guides group напишет

---

## 5. Проверка доступности

- Смотрит **и сайт и calendar** — потому что canceled booking может не освободить лодку на сайте → нужна двойная проверка вручную

### Боль: нет единой системы availability
- Отмена на сайте не всегда корректно освобождает слот
- Нужно сверять два источника каждый раз

---

## 6. Ошибки

- **Дата:** основная ошибка — перепутал месяц/неделю (особенно с Klook — US date format)
- Решение: взаимопроверка смен (если с Yuda — проверяют друг друга)
- Не было случаев неправильной лодки или цены

---

## 7. Работа с отменами / возвратами

- OTA cancellation: проверяет booking reference в Kommo → confirm → cancel → уточняет у гостя если сомнения
- Website cancellation: шаблон подтверждения → возврат от accounting → proof screenshot гостю
- Date change: проверяет доступность → меняет лодку → новое подтверждение чатом + email
- Рефанд при настаивании: стараются придерживаться политики → эскалирует к Диме/Кариму
- При настойчивых гостях → предлагает Flexible Booking как альтернативу полному refund

---

## 8. Работа с отменами туров по погоде

- Шаблоны «cancelled today» / «cancelled tomorrow»
- С коллегой делят: один делает shared, другой — privates
- Widia иногда приносит ответы гостей из офиса (кто хочет рефанд, кто рескедж)
- Всё равно ждут текстового подтверждения от гостя
- ~5-10 минут на 100+ гостей при совместной работе
