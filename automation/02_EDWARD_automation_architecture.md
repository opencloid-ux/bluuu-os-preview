## Кто такой Эдвард в системе

Эдвард — **фронтлайн GR**. Смена: обработка входящих запросов, ввод OTA-букингов, проверка availability, подтверждение туров, координация транспорта и vendor boats. Работает с Yuda — проверяют друг друга при вводе дат. Главная специализация: OTA-обработка и операционная точность.

**Главная боль:** OTA-букинги вводит вручную каждый день → ошибки с датами (особенно Klook US-формат). Потерял Kommo-виджет платёжных ссылок → теперь ручная работа на отдельном сайте.

## Архитектура автоматизации

### Модуль 1: OTA Auto-Import

**Проблема:** Эдвард вводит OTA-букинги вручную → ошибки дат (особенно Klook US-формат mm/dd vs dd/mm).

```
OTA Платформы (Viator / Klook / GYG / Airbnb / GetMyBoat / ClicksBoat)
         ↓
[API Integration Layer] — тянет новые букинги каждые 15 мин
         ↓
[Parser] — нормализация форматов дат (US → ISO), языков, полей
         ↓
[Odoo/Kommo] — автоматическое создание лида с полями:
  - Гость: имя, контакт, язык, гражданство
  - Тур: дата (нормализованная), тип, количество pax
  - Экстры: dietary, transfer, add-ons
  - OTA reference number
         ↓
[Notify Edward] — «New booking: Klook, 3 pax, SD boat, Apr 15»
  → Эдвард review и confirm (не вводит данные, только проверяет)
```

**Результат:** Эдвард с 60 мин/день ввода → 10 мин проверки. Ошибки дат = ноль.

**Klook date format fix:** отдельное правило парсера — Klook US-формат → всегда конвертировать в ISO 8601.

### Модуль 2: Unified Availability Dashboard

**Проблема:** Эдвард проверяет availability в двух местах (сайт + Calendar) потому что отмена не всегда освобождает слот на сайте.

```
[Single Source of Truth: Odoo Availability Engine]
├── Каждая лодка: capacity, тип, дата, статус
├── При любом событии (букинг / отмена / перенос):
│   → обновляется во всех подключённых каналах одновременно:
│     - Сайт bluuu.tours
│     - OTA (через Channel Manager API)
│     - Internal Calendar (Odoo)
│     - Dashboard Aline / Эдварда
└── Нет больше «два источника» — один
```

**Dashboard для Эдварда:**
```
[Сегодня — 07 апр]
```

### Модуль 4: Transport & Vendor Auto-Coordination

**Проблема:** Эдвард вручную пишет в WhatsApp → G. David (транспорт) + vendor boats.

```
T-24ч до тура:
[Odoo] → автоматически формирует:
  1. Transport brief для водителя:
     - Гость: имя, телефон, отель
     - Pickup: время, место
     - Маршрут: Серанган
  2. Vendor boat brief:
     - Лодка, pax count, dietary, extras
     - Время отправления

  → Отправляет через WhatsApp/email:
     - Водителю (через Respond.io)
     - Vendor boat captain
  → Эдвард получает confirmation статус в дашборде
```

**Утренний check T-4ч:**
```
[Auto-check at 06:00]:
  → Все водители подтвердили? ✅/❌
  → Все vendor boats готовы? ✅/❌
  → Если ❌ → alert Эдварду немедленно
```

### Модуль 5: Weather Live Feed

**Проблема:** Эдвард ждёт когда кто-то напишет погоду в 6:30. Нет автоматики.

```
[Weather Bot — 06:00 ежедневно]:
  → Тянет данные по Серангану из BMKG API / Windy API:
    - Высота волн (метры)
    - Скорость ветра (knots)
    - Видимость
  → Сравнивает с пороговыми значениями:
    - Волны < 1.5м AND ветер < 20 knots → ✅ GO
    - Волны 1.5-2м OR ветер 20-25 → ⚠️ CAUTION (David + Geebran решают)
    - Волны > 2м OR ветер > 25 → 🔴 NO-GO (все отменяются автоматически)
  → Публикует в GR group + Guides group + David
  
При NO-GO:
  → Триггер: mass cancellation flow (шаблоны для всех гостей дня)
```

### Модуль 8: AI Draft для сложных гостей

**Проблема:** Эдвард сейчас уходит в ChatGPT лично при агрессивных гостях — не интегрировано.

```
В интерфейсе лида:
  → Кнопка «🤖 AI Draft Reply»
  → AI анализирует:
    - Историю диалога с гостем
    - Тип проблемы (жалоба / запрос refund / агрессия)

  → Генерирует черновик ответа:
    - Empathetic opening
    - Факты из данных тура
    - Предложение решения (в рамках политики)
  → Эдвард редактирует и отправляет
```
