## Кто такая Widia в системе

Widia — **фронтлайн физического опыта гостей**. Она — первое лицо компании для всех гостей в офисе Серангана. Встречает, делает check-in, решает проблемы на месте. Под неё замыкается всё что происходит на берегу: confusion shared/private, last-minute изменения, погодные отмены, опоздавшие гости, жалобы в моменте.

**Главная боль:** ручной чекин с бумажным списком → путаница shared/private → гости не знают на какую лодку идти → хаос при большом потоке. Плюс: погодные отмены без предупреждения — узнаёт последней.

| Время | Задача | Боль |
|-------|--------|------|

| 06:30 | Гости начинают приходить | Нет QR-скана → ищет в списке вручную |

| Внезапно | Отмена по погоде | Узнаёт последней → гости уже в офисе |
| В течение дня | Жалобы в моменте | Нет доступа к данным гостя → не знает историю бронирования |

## Архитектура автоматизации

### Модуль 1: Digital Check-in

**Главный запрос из интервью: цифровой список + QR-скан.**

```
[Check-in Dashboard — Widia's iPad/Tablet]:

Вид экрана (07:00–08:30):
┌─────────────────────────────────────────────────────────┐
│  BLUUU TOURS — Check-in — 07 апреля 2026               │
│                                                         │
│  SD Boat (Classic Shared) — 16 pax expected            │
│  ░░░░░░░░░░░░░░░░ 0/16 checked in                      │
│  [John Smith]    [Jane Smith]    [Carlos M.]            │
│  [Maria G.]      [Tom W.]        [Alice K.]  ...        │
│                                                         │
│  MY Darling (Private) — 8 pax                          │
│  ░░░░░░░░░░░░░░░░ 0/8 checked in                       │
│  [The Johnson Family — 4 adults 2 kids]                 │
└─────────────────────────────────────────────────────────┘

Гость подходит:
  Опция A: QR-скан (гость показывает QR из confirmation email/WA)
    → система находит бронирование → ✅ checked in
  Опция B: Widia нажимает имя в списке → ✅ checked in
  Опция C: Walk-in / ищет по имени → ручной поиск в поле

При check-in:
  → Widia видит: [John Smith] — SD Boat — Classic Shared — Seat 3
  → Dietary: 🥗 Vegetarian
  → Extras: 🤿 Snorkeling equipment confirmed
  → Хotel: Kuta Paradiso (водитель доставил)
```

**Результат:** нет путаницы shared/private — каждый гость привязан к лодке.

### Модуль 2: Shared vs Private — визуальная навигация

**Проблема из отзывов: гости думали что у них Private, оказалось Shared.**

```
[Physical + Digital разграничение в офисе]:

1. Цифровые дисплеи у каждой лодки (или A4 печать):
   «SD BOAT — Classic Shared Tour — 16 pax total»
   «MY DARLING — Private — Johnson Family»

2. При check-in система показывает Widia:
   Цвет лодки: 🔵 Shared / 🟡 Premium Shared / 🟢 Private
   → Widia сразу видит и объясняет гостю правильно

3. Confusion detection:
   Если гость говорит «я на private» но бронирование Shared:
   → Widia нажимает «Dispute» → система показывает детали бронирования
   → Eсли реальная ошибка → Alert Edward/Дима немедленно
```

### Модуль 3: Weather Alert — Early Warning

**Проблема: Widia узнаёт об отмене по погоде последней — гости уже в офисе.**

```
[Weather Decision Timeline]:

05:30 — Data collection (BMKG + Windy API)
05:45 — David (Head of Fleet) + Geebran принимают решение GO/NO-GO
05:46 — Если NO-GO → НЕМЕДЛЕННО:
  → Widia: push-уведомление на планшет/телефон
    «🔴 NO-GO TODAY — All tours cancelled»
  → GR team: авто-рассылка гостям (WhatsApp / email / OTA-messaging)
  → Гости с shuttle: уведомление водителям «не забирать»

05:47–06:30 — Widia готова:
  → Если гости всё равно приходят (не получили уведомление):
    → Планшет показывает скрипт: что говорить, как предложить перенос / рефанд
    → One-click: зафиксировать гостя как «came despite cancellation»
      → автоматически создаёт rescheduling задачу в GR
```

### Модуль 4: Widia как Operations Hub

**Widia видит гостей физически — ценный источник данных, сейчас не используется.**

```
[Real-time Updates от Widia]:

При check-in — Widia вводит фактические данные:
  ├── Pax count (фактический vs бронирование → alert если расхождение)
  ├── «2 детей, возраст 5 и 7» — важно для ретрибуции
  ├── Dietary изменения на месте («мой муж оказывается не ест рыбу»)
  └── Физическое состояние гостей («один явно морская болезнь»)

Эти данные автоматически уходят в:
  → Geebran (гид на борту знает заранее)
  → Ресторан (обновлённые dietary)
  → Ретрибуция (фактическое количество детей)
```

### Модуль 5: Complaint Management на берегу

**Жалобы в моменте — Widia разбирается без данных о бронировании гостя.**

```
[Guest Profile — быстрый доступ]:

Widia подходит к расстроенному гостю:
  → Открывает: «Найти гостя» на планшете → сканирует QR или вводит имя
  → Видит ПОЛНУЮ КАРТОЧКУ:
    - Имя, контакт, OTA, тип тура
    - История (был ли раньше, были ли жалобы)
    - Текущий гид, лодка
    - Комментарии из GR

При конфликте:
  → Widia: кнопка «Эскалировать» → Дима/Geebran получают alert с именем гостя
  → Widia видит скрипт: что говорить в данной ситуации (по типу жалобы)
  → Фиксирует жалобу: тип → автоматически в систему (не звонок, не чат)
```

### Модуль 6: Shuttle Bus Integration

**Проблема из отзывов (2026): гость опоздал на 1 минуту, шаттл уехал.**

```
[Shuttle Bus Tracker]:

Каждый шаттл:
  → GPS-трекинг в системе
  → Widia видит на планшете: где каждый шаттл, когда прибудет

Гость не сел на шаттл:
  → Водитель отмечает «guest missing» в app → Widia alert
  → Widia: найти в списке → позвонить гостю (один клик на звонок)
  → Если гость уже едет → координировать встречу на Серангане напрямую

При NOGO:
  → Все шаттлы получают «Return to base» через систему
  → Гости в шаттлах получают SMS/WA: «Tour cancelled, driver will return you»
```
