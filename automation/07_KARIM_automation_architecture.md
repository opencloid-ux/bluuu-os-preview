## Кто такой Карим в системе

Карим — **единая точка истины для всего что касается денег и стратегии**. Operations Director + Finance Director в одном лице. Все ценовые решения — через него (10-20 запросов/день). Все рефанды выше порога — через него. Все стратегические эскалации — через него. Он читает из 5+ WhatsApp чатов одновременно и принимает решения по оперативным вопросам вручную.

**Главная системная проблема:** Карим — боттлнек. Если он недоступен → ценообразование стоит, рефанды висят, операционные решения зависают.

## Архитектура автоматизации

### Модуль 1: Pricing Engine — убираем Карима из цикла

**Проблема:** Ценообразование живёт в голове Карима. 10-20 запросов/день = Карим недоступен для стратегии.

```
[Pricing Rules Engine]:

Карим один раз (!) настраивает правила в системе:
  ├── Base price matrix:
  │   - Tour type × Boat tier × Season
  │   - Essential / Comfort / Premium
  │   - High season (Apr-Oct) / Low season (Nov-Mar)
  ├── Dynamic pricing rules:
  │   - Заполненность > 50% за 7 дней → Tier 2
  │   - Заполненность > 80% за 3 дня → Tier 3
  │   - Заполненность < 30% за 2 дня → Flash discount (опционально)
  ├── Group discounts: >8 pax / >12 pax
  ├── Agency rates (фиксированные по контракту)
  └── Exceptions: только Карим может ручную цену выставить

Менеджеры:
  → Открывают систему → цена уже рассчитана автоматически
  → Не пишут Кариму → 0 запросов вместо 10-20 в день

При нестандартной ситуации:
  → Менеджер нажимает «Custom price request»
  → Карим получает push → approve/reject одним нажатием
  → История всех кастомных цен с обоснованием
```

**Результат: Карим освобождается от 10-20 запросов/день = +2 часа/день для стратегии.**

### Модуль 3: Refund Approval Workflow

**Сейчас: рефанды висят пока Карим не ответит в WhatsApp. Нет структуры, нет трекинга.**

```
[Refund Workflow]:

Менеджер инициирует рефанд:
  → Указывает: гость, сумма, причина, OTA
  → Система автоматически:
    - Если сумма < X IDR → auto-approve (в рамках политики)
    - Если refund-for-review → эскалация Кариму
    - Если > порога → push Кариму с деталями

Карим:
  → Видит очередь pending refunds с контекстом
  → Approve / Reject / Counter-offer → одно нажатие
  → НЕ нужно искать историю в WhatsApp

Dashboard:
  ├── Total refunds this month: X IDR
  ├── Refund-for-review spend: Y IDR
  ├── Avg refund rate by OTA
  └── Top причины refund
```

### Модуль 4: Low Season Protection

**Критическая проблема: три убыточных месяца. Нет резервного фонда, нет системы.**

```
[Seasonal Cash Flow Management]:

Forecast module:
  → На основе исторических данных (2022-2025) + текущих темпов
  → Прогнозирует выручку на 90 дней вперёд
  → Alert: «Через 45 дней ожидается low season. На основе 2025 — дефицит 400M IDR»
  → Карим видит заранее → принимает решения:
    - Когда начать накапливать резерв
    - Когда запустить low-season offers
    - Когда сократить переменные расходы

Low Season Playbook (для автоматизации):
  → Октябрь: auto-trigger — запустить «Early Bird» для ноябрь-марта
  → Ноябрь: flash deals через Klook (высокая конверсия в low season)
  → Январь-март: Corporate/B2B рынок (конференции, тимбилдинги, стабильный спрос)
```

### Модуль 5: Channel Profitability — убрать плохие каналы

**Из интервью Димы: Headout и TripCom — самые проблемные. Из финансов: Instagram ROAS = 0.04x.**

```
[Channel Profitability Dashboard]:

По каждому каналу в реальном времени:
  ├── Gross revenue
  ├── Commission rate
  ├── Net revenue
  ├── Cancellation rate (этот канал)
  ├── Average complaint rate
  ├── ROAS (если платный трафик)
  └── Net margin

Карим видит:
  Headout: Net margin -2.3% (комиссия + рефанды превышают выручку)
  → Decision: закрыть или пересмотреть условия

  Instagram Ads: ROAS 0.04x
  → Decision: перенести бюджет в Google (ROAS 1.8x)

Weekly alert: «Channel [X] below breakeven threshold — review needed»
```

### Модуль 6: Operations Alert Hub (убираем Карима как роутер)

**Сейчас: Карим — точка маршрутизации всех операционных вопросов. Это не масштабируется.**

```
[Smart Escalation Rules]:

Уровень 1 (менеджер решает сам):
  → Стандартные запросы в рамках политики
  → Рефанды < порога
  → Изменения дат (если есть доступность)

Уровень 2 (автоматический → Дима):
  → Нестандартный рефанд
  → Конфликтный гость (агрессия)
  → OTA-жалоба требует ответа

Уровень 3 (автоматический → Карим):
  → Рефанд > X IDR
  → Overoverbooking
  → Потеря клиента (повторный гость, высокий LTV)
  → Финансовые решения (новый контракт с агентом)
  → Репутационный риск (потенциально вирусный отзыв)

Карим:
  → Получает только Level 3 alerts
  → Каждый с контекстом: история, рекомендация AI, сумма
  → Approve / Reject / Delegate — одно нажатие
```
