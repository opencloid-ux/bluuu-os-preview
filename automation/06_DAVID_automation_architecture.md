## Архитектура автоматизации

### Модуль 1: Auto-Passenger List для Bandar

**Самая болезненная ежедневная задача David из интервью.**

```
[T-1 день, 20:00 — Auto-Generate Passenger Manifest]:

Источник данных: Odoo (все бронирования на завтра)
  ├── По каждой лодке: список гостей
  │   - Имя (как в паспорте — из OTA/booking)
  │   - Гражданство
  │   - Количество детей/взрослых
  └── Статус: confirmed / pending

Система генерирует:
  → PDF Passenger Manifest (формат bandar)
  → По каждой лодке отдельно
  → Автоматически отправляет:
    - David: для проверки и подписи (цифровая или физическая)
    - Офицеру bandar (WhatsApp / email — если принимают)
    - Архив: в Odoo по дате

При изменениях (добавление гостей, отмена):
  → Manifest обновляется автоматически
  → David получает alert: «Manifest updated: SD Boat +2 pax, total 14»
  → Один клик: re-send updated manifest к bandar

Утром (06:00):
  → David проверяет финальный список на планшете
  → Confirm → лодки выходят
```

**Результат:** 30-45 мин → 5 мин проверки + confirm.

### Модуль 2: Captain Scheduling — Digital

**Проблема: физическое расписание → теряется, нет истории, нет аналитики.**

```
[Captain Scheduling System]:

David формирует расписание в системе (вечером):
  ├── Перетаскивает капитанов на лодки (drag & drop)
  ├── Система предупреждает:
  │   - Капитан работает 6 дней → нужен выходной
  │   - Капитан не имеет допуска на эту лодку
  │   - Капитан уже назначен на другую лодку в этот день

Капитаны получают:
  → Уведомление в WhatsApp/app: «Tomorrow: SD Boat, 07:00 departure, 16 pax»
  → Подтверждают: ✅

David видит в dashboard:
  [07 апреля]
  SD Boat:    Captain Wayan ✅
  Jayanadi:   Captain Made ✅
  MY Darling: Captain Putu ⏳ (не подтвердил)
  → Auto-reminder Putu, если нет ответа → alert David

История расписания:
  → Кто как часто на каких лодках
  → Overtime tracking
  → Availability для будущего планирования
```

### Модуль 3: Preventive Maintenance Engine

**312 инцидентов с двигателями за 4 года. Нет системных алертов = реактивный ремонт вместо превентивного.**

```
[Каждая лодка — цифровой паспорт]:
  ├── Engine hours (фактические, обновляются после каждого рейса)
  ├── История ТО: дата, тип (100h/300h/400h), что заменено
  ├── Открытые issues (что сломано, кем замечено)
  └── Следующее плановое ТО: дата/часы

[Auto-Alert система]:
  
  Engine hours → 90h: «SD Boat approaching 100h service — schedule now»
  Engine hours → 95h: «SD Boat 100h service OVERDUE in 5h — confirm appointment»
  Engine hours → 280h: «SD Boat approaching 300h major service»

  Компоненты с датами (не только часы):
  → Impeller: каждые 6 месяцев → alert T-14 дней до
  → Oil: каждые 100 часов → совмещать с 100h service
  → Belts: каждые 400h → alert T-20 дней

[David → планирует ТО]:
  → Видит все pending ТО на следующие 30 дней
  → Выбирает день (когда лодка не в расписании)
  → Назначает механика
  → ТО выполнено → механик закрывает в системе + hours reset

При экстренном ремонте:
  → Поломка зафиксирована в системе
  → Автоматически убирает лодку из расписания
  → Триггер: поиск замены (из списка доступных лодок)
  → Alert Geebran (нет лодки → нужна замена гидов)
```

### Модуль 4: Fuel Tracking

**Сейчас: ручной журнал per boat per day → в accounting.**

```
[Fuel Log — Digital]:

Каждый капитан после рейса:
  → В app: литры заправлено + одометр (engine hours)
  → Фото чека (если заправка не на базе)

Dashboard David:
  ├── Топливо сегодня по всем лодкам
  ├── Средний расход per hour по типу лодки (benchmark)
  ├── Аномалии: «MY Darling сегодня +40% сверх нормы» → alert
  └── Месячный отчёт → авто в accounting

Аномалии топлива:
  → Повышенный расход = возможная проблема двигателя → сигнал на ТО-проверку
  → Нулевое заправление при рейсе = капитан не заполнил → reminder
```

### Модуль 5: Incident Management

**Из FINAL_ANALYSIS: 312 инцидентов с двигателями — все реактивные. Нет трекинга паттернов.**

```
[Incident Reporting]:

Капитан/механик фиксирует инцидент в app:
  ├── Тип: Engine / Equipment / Safety / Guest / Route
  ├── Лодка + дата + engine hours
  ├── Описание (кратко + фото если есть)
  └── Severity: Minor / Major / Emergency

David видит:
  → Все открытые инциденты по лодкам
  → Паттерны: «SD Boat — 4 engine incidents in 2 months → deep inspection needed»

Analytics (для David + Alex):
  → Инциденты по типу / по лодке / по месяцу
  → MTBF (Mean Time Between Failures) по каждой лодке
  → Cost of repair vs remaining boat value → помогает решению о замене флота
```

