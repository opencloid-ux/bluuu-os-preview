## 2. Ключевые операционные паттерны

### 2.1 Ежедневный клиренс (критический процесс)

**Что происходит каждое утро:**

1. Капитаны фотографируют бумажные пассажирские манифесты (списки)
2. Отправляют фото в WhatsApp-группу клиренса (Gohin Island Clearance / SRG-DTB-MKP)
3. Береговой служащий (BUMDA / Syahbandar) видит фото и даёт разрешение
4. Лодка уходит

**Пример из SRG_DTB_X_MKP:**
```
[время] Yande DTB: image omitted × 3
[время] Captain Sweet Heart 3: image omitted × 3 — "Sweet heart 4 tujuan serangan"
[время] Captain RJ: "Riki J tujuan serangan 🙏" + image omitted × 3
```

**Участники:** I Wayan Sukarja Kudus (клиренс Нуса-Пенида), Yoga Gohin NP, капитаны всех лодок включая конкурентов (Sanjaya 1&2, Sea Dragon, Wahana Baruna, Haruku, Jayanadi, Little Star).

**Проблемы:**
- 100% ручной процесс — фотографируют физические бумаги ежедневно
- Пассажирский список в боне ≠ пассажирский список в клиренсе (могут расходиться)
- Нет цифровой записи: кто уехал, на какой лодке, когда
- Если нет интернета или фото плохое — задержка отправления

### 2.2 Охрана лодок (Boat Security)

**Паттерн:** Wayan Sutana (охранник Serangan) отправляет 4 фото × 3 раза в день:
- ~03:30–04:00 — ночной обход
- ~06:00–06:10 — утренний обход (перед отправлением)
- ~20:00–21:00 — вечерний обход (после возвращения)

**Пример:**
```
[01/06/24, 6.05.06 AM] Wayan Sutana Boat Security Serangan: image omitted × 4
[01/06/24, 6.14.00 AM] David: 🙏🏾
```

**Вывод:** David получает 12 фото в день, смотрит, ставит 🙏, двигается дальше. Нет structured log, нет alert при аномалии, нет digital timestamp.

### 2.3 Daily Logistic Report (главный оперативный чат)

Формат сообщений — вольный текст:
- Количество пассажиров по лодкам
- Время отправления/прибытия
- Топливо (BBM) — сколько залили, в какую лодку
- Инциденты (двигатель, волны, задержка)
- Координация транспорта (водителей)

**Типичные записи:**
```
[дата] Kapten: "Berangkat jam 7.30 pax 22 orang"
[дата] David: "BBM sudah isi 100L untuk Angels"
[дата] Kapten: "Mesin ada masalah, jam 9 baru berangkat"
```

**Проблема:** 52,000 строк свободного текста. Данные есть, но не структурированы. Нельзя автоматически посчитать: сколько всего пассажиров за день, топливо за неделю, задержки по лодке.

## 3. Системные боли (Fleet Operations)

### 🔴 Критические

**Боль 1: Пассажирский манифест — 100% ручной и бумажный**
- Каждое утро капитан заполняет бумажный список, фотографирует, отправляет в WhatsApp
- Нет digital database пассажиров по лодкам
- Данные из CRM (Odoo) не попадают к капитану автоматически
- Ошибки неизбежны: pax в брони ≠ pax на борту
- Из интервью David: «Passenger list для bandar — ручная работа каждый день» — главная боль

**Боль 2: Расписание капитанов — бумага**
- Распечатывается физически, раздаётся вручную
- Изменения → новая распечатка или голосовое сообщение
- Капитан не знает о смене расписания пока David лично не скажет

**Боль 3: Нет единого флот-дашборда**
- David мониторит 10+ WhatsApp-групп одновременно
- Статус каждой лодки: вышла? вернулась? инцидент? — только из чата
- Нет сводной картины в реальном времени

### 🟠 Высокие

**Боль 4: Топливо — ручной учёт**
- Сколько залили, в какую лодку, когда — всё в виде текстовых сообщений
- Нет базы данных. Нельзя посчитать расход за месяц без ручного парсинга
- Аномалии (перерасход на лодке) не видны без ручного анализа

**Боль 5: Техническое обслуживание без алертов**
- Интервалы ТО: 100/300/400 часов — отслеживаются вручную (или не отслеживаются)
- Из интервью David: 312 инцидентов с двигателями зафиксировано в данных
- Нет проактивных напоминаний: «Sweet Heart 4 — через 15 часов ТО»

**Боль 6: Охрана — фото без структуры**
- 12 фото в день, David смотрит и ставит 🙏
- Нет digital log, нет alert при аномалии
- Нельзя сказать: «В ту ночь охранник не обходил Angel Boat» без парсинга чата

**Боль 7: Клиренс Нуса-Пенида — фото манифестов в WhatsApp**
- Конкурентная группа (другие операторы видят расписание Bluuu)
- Документы фотографируются каждый день вместо авто-генерации
- При плохом соединении — задержка отправления

### 🔵 Средние

**Боль 8: Porт-слоты — ручные запросы в WhatsApp**
- Нестандартное время → David пишет Nyoman лично
- Зависит от личных отношений, не от системы
- Нет подтверждения в виде документа

**Боль 9: Check-in reporting — свободный текст**
- Нельзя отчитать кто опоздал, какие отправления задержались
- Нет автоматического алерта если капитан не отметился к 06:00

**Боль 10: Погода — ручное распространение**
- Данные из внешних источников нужно вручную копировать в чат
- Нет авто-интеграции с wind/wave API (как в интервью David: «Rule of Two No's»)

## 6. Связь с другими командами (кросс-команда)

### Fleet ↔ GR
- Пассажирский список из Odoo (создан GR) должен автоматически идти к David
- Изменения брони (отмены, переносы) меняют список → сейчас David узнаёт с задержкой
- **Решение:** Odoo webhook → авто-обновление манифеста при изменении

### Fleet ↔ Guides (Geebran)
- Гид назначен на тур → должен знать: какая лодка, время, pax — из системы, не из WhatsApp
- SweetHeart4 чат показывает: Geebran и David координируются вручную по каждому туру
- **Решение:** единый дашборд «тур дня» с гидом + лодкой + пассажирами

### Fleet ↔ Finance (Karim)
- Топливо, ТО, операционные расходы — всё в свободном тексте сейчас
- Karim добавлен в Boat_Security группу — сигнал, что финансовый контроль важен
- **Решение:** структурированный fuel/maintenance log с экспортом в бухгалтерию

## 8. Стратегический вывод

Команда David — **наиболее „аналоговая" из всех команд Bluuu Tours**.

GR команда имеет CRM, боты, OTA-кабинеты — цифровая инфраструктура есть, просто плохо связана.

Fleet Operations работает практически полностью через:
- WhatsApp-сообщения
- Физические документы (бумажные манифесты)
- Личные звонки и договорённости

**Потенциал цифровизации — максимальный.** При этом:
- Технические ресурсы команды ограничены (капитаны → простые инструменты)
- Любое решение должно быть: мобильное, очень простое, на Bahasa Indonesia
- Интеграция с Odoo (GR) — обязательна для auto-manifest

**Порядок внедрения:**
1. Quick wins (Google Forms/Sheets) — немедленно, без разработки
2. Fleet Dashboard (Odoo + мобильное приложение) — Месяц 2–3
3. Auto manifest → clearance — Месяц 3 (требует API-интеграции с booking engine)
4. Maintenance tracking с алертами — Месяц 2 (часть Fleet Management модуля)
